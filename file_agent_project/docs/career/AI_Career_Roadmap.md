## 30 天计划

### 第 1 周：把当前 File Agent 做完整

目标：
- 把当前项目从“能跑”提升到“像项目”

要做：
- 完善 `loop_count / retry_count` 控制
- 给未知 `tool_name` 增加兜底
- `ask_user` 不再硬编码 `file_name`
- 工具结果回传模型更稳定
- 把 API key 从代码里迁移到环境变量
- 完善 README

交付物：
- 一个能稳定运行的本地文件 Agent
- 一份清晰 README
- 一份迭代问题清单

---

### 第 2 周：做一个最小 RAG 项目

目标：
- 补上“知识型 AI 应用”能力

项目建议：
- 本地文档问答系统
- 或企业知识库问答 Demo

最小功能：
- 文档加载
- 文档切分
- 向量化
- 检索
- 检索结果增强回答

要理解：
- RAG 不等于 Memory
- 文档切分为什么影响效果
- Prompt 如何限制“只根据检索结果回答”

交付物：
- 一个最小 RAG 问答项目
- 一张流程图：文档 -> 切分 -> 向量化 -> 检索 -> 回答

---

### 第 3 周：做一个业务型 Agent

目标：
- 让项目更贴近真实岗位

建议方向：
- 简历分析助手
- 工单分类与总结助手
- 代码解释/接口文档生成助手

要求：
- 有明确业务输入
- 有明确业务输出
- 最好有一个工具或外部数据源
- 最好有简单多轮逻辑

交付物：
- 一个能讲清业务价值的 AI 项目

---

### 第 4 周：整理简历和面试材料

目标：
- 把“会做”变成“会表达”

要做：
- 整理 3 个项目描述
- 每个项目写清楚：
  - 背景
  - 目标
  - 技术方案
  - 难点
  - 你的职责
  - 效果
- 准备常见面试题答案

重点面试题：
- Agent 和普通聊天系统的区别
- Tool Calling 怎么设计
- State / Memory / RAG 怎么区分
- Prompt 如何约束结构化输出
- 工具调用失败如何处理
- Orchestrator 为什么重要
- Java 系统怎么接入 AI 能力
