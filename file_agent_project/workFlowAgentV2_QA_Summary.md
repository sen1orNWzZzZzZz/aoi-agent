# WorkflowAgentV2 问答总结

## 目的
这份文档用于总结 `workFlowAgentV2` 版本中已经建立起来的核心工程理解，帮助后续整理 README、流程图和面试表达。

## 1. Action 和 ToolResult 分别是什么
- `Action` 是模型输出的下一步动作，表示系统接下来应该做什么。
- `ToolResult` 是工具执行后的结果，表示这一步实际发生了什么。
- 两者不能混在一起，因为一个面向决策，一个面向执行反馈。
- 如果混用，系统会分不清“这是模型在下指令”还是“这是程序在回传观察结果”。

## 2. history 和 state 的区别是什么
- `history` 是语义上下文，主要给模型看，帮助模型理解用户说过什么、之前执行过什么。
- `state` 是结构化上下文，主要给程序看，用来控制 workflow 的阶段、队列和状态推进。
- 不能只靠 `history`，因为模型更擅长理解文字，不擅长稳定维护严格流程。
- 要让程序可靠地推进多步任务，必须把关键流程信息落到结构化的 `state` 中。

## 3. 为什么多文件任务要尽快落到 pending_files
- 多文件任务一旦识别出来，就应该尽快把文件列表写入 `pending_files`。
- 因为文件顺序、完成进度、失败重试这类事情，属于程序擅长处理的确定性流程。
- 如果只把这些信息留在 `history` 里让模型自己记，模型就要同时承担语义理解和流程记忆，系统稳定性会下降。
- 把文件列表落到 state，orchestrator 才能按队列逐个消费，workflow 才真正可控。

## 4. task_type 和 workflow_type 分别代表什么
- `task_type` 是模型输出的任务识别结果，用来表达模型认为当前任务是什么。
- `workflow_type` 是程序写入 state 的流程控制字段，用来表达系统当前实际处于哪个 workflow。
- `task_type` 交给模型，是因为自然语言任务识别属于语义理解问题。
- `workflow_type` 交给 orchestrator，是因为流程推进、阶段切换、失败恢复都需要程序做确定性控制。

## 5. 三类 workflow 在什么时候算任务完成
- `single_file_read`：当文件读取完成，并把读取结果作为最终回复返回给用户时，任务完成。
- `single_file_summary`：当文件读取完成后，还需要进入总结阶段；只有模型基于文件内容生成最终总结并返回给用户时，任务才完成。
- `multi_file_summary`：当所有文件都读取完成后，还需要基于 `collected_contents` 进入最终总结阶段；只有总结结果返回给用户时，任务才完成。
- 工具执行成功不等于任务结束，尤其是 summary 类任务，读取文件只是拿到原始材料，后面还需要总结阶段。

## 6. ask_user 时为什么通常保留 workflow_type
- 当需要 `ask_user` 时，workflow 通常并没有结束，而是进入等待用户补充信息的暂停状态。
- 如果这时直接把 `workflow_type` 重置为 `idle`，系统就会丢失当前任务上下文。
- 用户下一轮补充信息回来时，程序会很难继续原来的 workflow，只能把它当成一个全新任务重新理解。
- 所以对于需要继续推进的任务，`ask_user` 时通常应保留 workflow 状态。

## 7. 多文件 workflow 的关键字段分别承担什么职责
- `pending_files`：还没有处理的文件队列，orchestrator 按顺序消费它。
- `completed_files`：已经处理完成的文件列表，用来记录进度。
- `collected_contents`：已成功读取文件的内容集合，通常是 `文件名 -> 文件内容` 的映射，是后续总结阶段的主数据源。
- `current_file`：当前正在处理或最近一次处理的文件，用来标记当前执行焦点。

## 8. 为什么最终总结阶段要以 collected_contents 为主数据源
- `collected_contents` 是多文件 workflow 执行后沉淀下来的结构化成果，表示哪些文件已经成功读取以及它们的内容。
- 相比之下，`history` 混合了用户输入、模型回复和工具执行记录，语义更杂，不适合作为最终总结的主数据源。
- 以 `collected_contents` 为主，可以让程序明确控制总结输入边界，也更方便后续做裁剪、排序、过滤等扩展。

## 9. 为什么文件失败处理要由 orchestrator 决定
- 文件读取失败后的处理属于确定性流程控制问题，而不是语义理解问题。
- orchestrator 更适合根据错误类型、当前状态和重试策略来决定是重试、追问还是终止。
- 如果把这部分交给 LLM 自由决定，系统行为会变得不稳定，也会削弱 workflow 的可控性和可复现性。

## 10. 为什么这个项目已经不是脚本式工具调用，而是最小 Workflow Agent
- 它不是“调完工具就结束”，而是形成了一个最小 Agent 闭环：模型先输出 `Action`，程序执行工具得到 `ToolResult`，再把结果回流给模型继续决策。
- 在这个闭环基础上，系统又引入了 `orchestrator + state`，把 `workflow_type`、`pending_files`、`completed_files`、`collected_contents` 这类状态落到了程序里。
- 这样系统就能处理单文件读取、单文件总结、多文件总结，以及失败后的重试、追问和终止分支。
- 所以这个项目的重点不是“会调工具”，而是“把不稳定的模型能力收束成可控的流程能力”。

## 当前版本的已知限制
- `ask_user` 之后恢复原 workflow 的能力还不完整。
- 当前错误分类和恢复策略仍然较简化。
- 当前主要是 CLI 版本，尚未服务化。
