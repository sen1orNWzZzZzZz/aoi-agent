## 1. 学习路线总览

这一阶段的学习目标，不是立刻会用某个框架，而是先理解 Agent 的本质。

建议顺序如下：

1. 先看懂最小 Agent 的结构
2. 再理解 Agent 的闭环执行逻辑
3. 再区分 Prompt、Tool、Memory、State、RAG、Orchestrator
4. 再去看 LangChain、LangGraph、OpenAI Agents SDK 等框架

一句话总结：

`先理解“为什么这样设计”，再学习“框架怎么写”。`

## 2. 预备知识：从最小 Agent 代码开始

这一部分来自最开始的代码理解阶段。虽然还没进入正式课程，但这是后面所有课程的地基。

### 2.1 一个最小 Agent 至少有哪些部分

最小 Agent 的核心结构可以理解为：

- 用户输入
- 决策函数
- 工具注册表
- 工具执行函数
- 回复函数
- 历史记录

在练习代码里，这些概念分别对应：

- `SimpleAgent.decide()`：做规则判断
- `self.tools`：工具注册表
- `get_time / list_files / read_file`：工具函数
- `respond()`：串联决策、执行、输出
- `history`：保存问答记录

### 2.2 `Tool` 类是什么

`Tool` 不只是一个普通类，它代表“一个工具的结构化定义”。

它通常包含三部分：

- 工具名：这个工具叫什么
- 工具描述：这个工具是做什么的
- 处理函数：真正执行这个工具的函数

也就是说，`Tool` 的作用不是“直接干活”，而是把“工具的元信息”和“执行入口”打包在一起，方便 Agent 统一管理和调用。

### 2.3 `handler=get_time` 和 `handler=get_time()` 的区别

这是最重要的入门点之一。

#### `handler=get_time`

表示把函数本身存进去。

此时 `handler` 里保存的是：

- 一个函数对象
- 一个稍后可以再调用的工具入口

#### `handler=get_time()`

表示立刻执行函数，把执行结果存进去。

如果 `get_time()` 返回的是字符串，那么最后 `handler` 里存进去的就是一个字符串，而不是函数。

这会带来两个问题：

1. 工具会在初始化时就被执行，不是等用户提问时再执行
2. 后续 `tool.handler(argument)` 会报错，因为字符串不是函数

#### 核心理解

Agent 需要的是：

- 先注册工具
- 后选择工具
- 再执行工具

所以要存的是“函数本身”，不是“函数执行结果”。

一句话记忆：

`不带括号是函数本身，带括号是立刻执行。`

### 2.4 `Callable[[str], str]` 是什么意思

这是 Python 的类型注解。

它表示：

- 这是一个可调用对象
- 接收 1 个参数，参数类型是 `str`
- 返回值类型也是 `str`

等价理解：

```python
def func(x: str) -> str:
    ...
```

在工具系统里，这意味着：

`handler` 应该是一个“接收字符串、返回字符串”的函数。

### 2.5 为什么 `self.tools` 常常设计成字典

这是工具注册的最小实现。

用字典管理工具有四个重要好处：

1. 统一注册和管理工具
2. 可以按名字快速找到对应工具
3. 新增工具时容易扩展
4. 把“决策”和“执行”解耦

更本质地说：

`self.tools` 是一个最小工具注册表。`

当 Agent 决策出“要调哪个工具”时，就可以通过工具名从这个注册表里取出真正的执行入口。

### 2.6 `decide()` 和 `respond()` 的职责区分

这是 Agent 设计里非常重要的职责分离。

#### `decide()`

负责：

- 读用户输入
- 判断下一步该做什么
- 决定是直接回复，还是调用某个工具

它更像“大脑中的决策层”。

#### `respond()`

负责：

- 接收用户输入
- 调用 `decide()`
- 真正执行工具或生成普通回复
- 组织最终输出
- 保存历史记录

它更像“把决策执行下去的主流程”。

一句话：

- `decide()` 负责想
- `respond()` 负责做

### 2.7 `history` 是什么

`history` 通常保存的是：

- 用户说过什么
- Agent 回复了什么

它最大的作用是让 Agent 在多轮对话里不至于立刻失忆。

但要注意：

`history` 不等于真正完整的 memory。`

因为它通常只是原始记录，不会自动提炼重点，也不一定支持长期保存、检索和复用。

### 2.8 预备知识常见误区

#### 误区 1：以为工具就是函数本身

不完全对。

函数是工具的执行入口，但 Agent 中的 Tool 往往还包含：

- 名字
- 描述
- 参数规范

#### 误区 2：以为 `history` 就是 memory

不对。

`history` 更像聊天流水，`memory` 更像提炼后的可复用信息。

#### 误区 3：以为能调用一次工具就是 Agent

不对。

Agent 更关键的是：

- 能根据目标决策
- 能调用工具
- 能根据结果继续或停止

也就是完整闭环。

### 2.9 预备知识课后问答

#### Q1：为什么 `handler` 要保存函数本身，而不是执行结果？

答：

因为 Agent 要先注册工具，再在合适的时机动态调用它。  
如果保存的是执行结果，工具会在初始化时就运行，后面也无法再次按需调用。

#### Q2：为什么真正执行工具是在 `respond()` 阶段，而不是 `decide()` 阶段？

答：

因为 `decide()` 只是负责判断“要做什么”，  
真正“拿起工具做事”是在 `respond()` 或它调用的执行层里完成的。

#### Q3：为什么要用字典管理工具？

答：

因为 Agent 需要按名字动态选择工具。  
字典正好适合充当最小工具注册表，便于统一管理、查找、扩展。

## 3. 第一课：一个最小 Agent 的完整执行链路

### 3.1 本课目标

理解 Agent 的本质，不是“用户说一句，模型回一句”，而是一个围绕任务目标运行的闭环。

### 3.2 最核心的一句话

`用户提目标 -> 模型决定动作 -> 工具执行 -> 模型看结果 -> 结束或继续`

这就是最小 Agent 的执行链路。

### 3.3 完整执行流程

#### 第一步：用户输入任务

用户给出的通常不是代码命令，而是任务目标，例如：

- 帮我查北京天气
- 看下当前目录文件
- 读取这个文件并总结

#### 第二步：组装上下文

系统通常会把这些内容一起送给模型：

- 系统提示词
- 当前用户输入
- 可用工具说明
- 部分历史对话
- 当前任务状态

#### 第三步：模型做当前这一步决策

模型可能会输出两类内容：

1. 直接回答
2. 工具调用意图

注意：模型本身不直接执行工具，它只是决定“该用什么工具、参数是什么”。

#### 第四步：Orchestrator 解析模型输出

调度中心负责判断：

- 这是普通回答
- 还是工具调用
- 工具名是什么
- 参数是否合法

#### 第五步：执行工具

工具真正与外部世界交互：

- 查天气
- 读文件
- 调接口
- 搜索资料

这一步得到的是“工具执行结果”。

#### 第六步：把工具结果再喂回模型

这一步非常关键。

因为工具只负责执行，不负责：

- 理解结果
- 判断是否完成任务
- 决定下一步动作

这些还需要模型来做。

一句话：

`工具提供观察，模型负责解释和再决策。`

#### 第七步：判断结束还是继续

如果结果已经足够回答用户，就停止。  
如果还不够，就继续下一轮：

- 再调用工具
- 或换策略
- 或追问用户

#### 第八步：生成最终回复

最终回复不是简单把工具原始结果丢给用户，而是：

- 结合用户目标
- 结合工具结果
- 组织成人能看懂的自然语言

#### 第九步：写入历史/记忆/状态

系统可以把本轮的部分信息保存起来，供后续使用。

### 3.4 这节课最重要的结论

`Agent 的本质不是某个模型、某个工具、某个框架，而是一套围绕目标执行的闭环。`

### 3.5 常见误区

#### 误区 1：工具调用就是 Agent

不对。工具调用只是能力点，真正的 Agent 是“决策 + 执行 + 观察 + 再决策”的闭环。

#### 误区 2：工具结果就是最终回复

不对。工具结果往往只是原始数据，最终回复需要模型整理后交付给用户。

### 3.6 课后问答

#### Q1：为什么工具结果要再喂回模型？

答：

因为工具只负责执行，不负责理解任务全局。  
模型需要根据工具结果判断：

- 是否已完成任务
- 是否还需下一步
- 如何组织最终回复

#### Q2：为什么“会调用一次工具”还不等于“具备 Agent 能力”？

答：

因为单次调用只是一个动作，不代表系统具备：

- 自主决策
- 多步循环
- 结果评估
- 停止判断

#### Q3：为什么说工具是在给大模型补手脚，而不是补大脑？

答：

因为大模型本来就擅长理解、推理和决策，也就是“大脑能力”。  
工具的作用是让它能与外部环境交互，相当于补上手脚和外部感官。

## 4. 第二课：Prompt 在 Agent 中到底起什么作用

### 4.1 本课目标

理解 Prompt 不只是“人设文案”，而是 Agent 行为控制逻辑的重要组成部分。

### 4.2 最核心的一句话

`Prompt 是自然语言形式的控制逻辑。`

### 4.3 Prompt 在 Agent 中控制什么

#### 1. 模型是谁

例如：

- 你是一个天气助手
- 你是一个文件分析助手
- 你是一个会使用工具的 Agent

这一步不是单纯立人设，而是在缩小模型的行为范围。

#### 2. 模型的目标是什么

Prompt 要告诉模型：

- 当前任务成功的标准是什么
- 什么情况下算完成
- 什么情况下不能乱猜

#### 3. 模型能用哪些工具

如果不告诉模型有哪些工具，它就不知道自己能做什么。

Prompt 或工具说明需要明确：

- 工具名
- 工具用途
- 使用场景
- 参数含义

#### 4. 什么时候该调工具，什么时候不该调

这是 Prompt 中非常关键的一层规则，比如：

- 外部真实信息必须查工具
- 能直接回答的就直接回答
- 不知道时不要编造

#### 5. 工具结果回来后该怎么处理

Prompt 还会影响模型看到 observation 后的行为，例如：

- 是直接结束
- 还是继续下一步
- 是否需要解释异常

#### 6. 最终输出长什么样

例如：

- 用简洁中文回答
- 先给结论
- 不要暴露中间推理
- 工具失败时说明原因

### 4.4 为什么 Prompt 更像程序的一部分

因为它不只是调语气，而是在决定：

- 模型怎么思考
- 模型怎么做决策
- 模型怎样用工具
- 模型何时停止

所以：

- 形式上像配置
- 作用上像程序逻辑

### 4.5 常见误区

#### 误区 1：Prompt 只是让回答更像人

不对。  
在 Agent 系统里，Prompt 更重要的作用是控制行为。

#### 误区 2：Prompt 写一点角色设定就够了

不对。  
一个能工作的 Agent Prompt 至少应该覆盖：

- 角色
- 目标
- 工具说明
- 使用规则
- 输出规范

### 4.6 课后问答

#### Q1：Prompt 更像配置，还是更像程序的一部分？

答：

两者都有，但在 Agent 中更接近“程序逻辑的一部分”。  
因为 Prompt 直接影响系统行为，而不是只影响文案。

#### Q2：Prompt 写不好会出什么问题？

答：

常见问题包括：

- 该调用工具时不调用
- 不该调用时乱调用
- 明明不知道还硬答
- 工具结果回来后不会处理

## 5. 第三课：工具描述为什么决定模型会不会正确调用工具

### 5.1 本课目标

理解工具描述不是装饰，而是模型决定“该用什么工具、怎么用工具”的关键依据。

### 5.2 最核心的一句话

`工具描述是给模型看的 API 文档。`

### 5.3 为什么工具描述很重要

模型看不到你的 Python 函数内部实现。  
它只能根据你暴露给它的信息做判断：

- 工具名
- 工具说明
- 参数说明
- 使用边界

如果这些信息写得不好，模型就容易用错工具。

### 5.4 好的工具名应该具备什么特点

应该尽量做到：

- 含义清晰
- 边界明确
- 不容易和其他工具混淆

例如：

- `get_weather`
- `read_local_file`
- `search_product_docs`

会比：

- `tool_1`
- `query_api`
- `process_data`

更适合给模型使用。

### 5.5 工具描述不仅告诉模型“能做什么”，也要告诉模型“不能做什么”

这点很重要。

例如天气工具应该告诉模型：

- 适用于天气查询
- 不适用于新闻检索
- 返回的是天气数据

这样模型才不会乱用。

### 5.6 参数说明为什么同样重要

工具描述帮助模型“选工具”，参数说明帮助模型“正确使用工具”。

如果参数说明不清，模型就容易：

- 少传参数
- 传错字段
- 值写得不规范
- 把整句自然语言塞进去

### 5.7 常见误区

#### 误区 1：工具能跑起来就行

不对。  
Agent 场景里，工具不仅要“能跑”，还要“让模型知道什么时候、为什么、怎么跑”。

#### 误区 2：工具描述可有可无

不对。  
没有描述，模型很难稳定地选对工具和传对参数。

### 5.8 课后问答

#### Q1：为什么说工具定义不只是程序执行入口，还是能力说明？

答：

因为在 Agent 中，工具定义既要提供真正执行的入口，也要向模型解释：

- 工具能做什么
- 什么时候用
- 参数怎么填

#### Q2：工具描述不清最常导致什么问题？

答：

最常见的是：

- 选错工具
- 参数传错
- 该调用时不调用
- 不该调用时乱调用

## 6. 第四课：参数设计为什么会直接影响 Agent 稳定性

### 6.1 本课目标

理解 Agent 稳定性常常不是坏在“不会选工具”，而是坏在“参数乱传”。

### 6.2 最核心的一句话

`工具选择解决“用哪个”，参数设计解决“怎么用”。`

### 6.3 参数本质上是什么

参数是模型和工具之间的接口。  
可以把它理解成模型填写给工具的表单。

如果表单设计得不清楚，模型就容易填错。

### 6.4 好的参数设计有哪些原则

#### 1. 参数名清晰

比起：

- `query`
- `data`
- `input`

模型更容易理解：

- `city`
- `date`
- `file_name`

#### 2. 参数数量适中

参数太多会增加模型出错概率：

- 漏填
- 乱填
- 填不合理组合

#### 3. 必填参数明确

如果缺这个参数工具就没法工作，就必须明确要求模型提供。

#### 4. 能枚举就尽量枚举

例如单位：

- `celsius`
- `fahrenheit`

会比让模型自由生成“摄氏度、C、℃、华氏度”等各种写法更稳定。

#### 5. 不要让一个参数承载太多语义

例如不要把地点、时间、输出风格、额外要求全塞进一个 `query_text`。

拆清字段后，模型更容易传对，工具也更容易处理。

### 6.5 参数设计为什么和异常处理有关

如果参数清晰，错误就容易判断：

- 缺哪个字段
- 哪个字段格式错了
- 哪个值不合法

如果参数过于模糊，异常处理会非常困难。

### 6.6 常见误区

#### 误区 1：工具越万能越好

不一定。  
Agent 场景里，过于万能的工具往往意味着：

- 参数复杂
- 模型难传对
- 稳定性下降

#### 误区 2：只要工具描述清楚，参数随便设计也行

不对。  
参数设计本身就是稳定性的一大来源。

### 6.7 课后问答

#### Q1：为什么参数越模糊，模型越容易犯错？

答：

因为模型不知道到底该把哪些信息填进去，容易把整句自然语言或不必要内容一起塞进参数里。

#### Q2：为什么参数设计会影响工具失败后的恢复能力？

答：

因为参数清晰时，系统可以更容易判断：

- 哪个参数缺失
- 哪个参数非法
- 是该重试还是该追问用户

## 7. 第五课：什么时候继续循环，什么时候停止

### 7.1 本课目标

理解稳定 Agent 的关键，不只是会调用工具，而是“知道什么时候该停”。

### 7.2 最核心的一句话

`Agent 不是做得越多越好，而是该继续时继续，该停止时停止。`

### 7.3 应该继续循环的情况

以下情况通常应该继续：

1. 当前结果不足以回答用户问题
2. 工具结果提示还需要下一步动作
3. 工具失败但仍存在合理重试空间
4. 任务本身就是多步任务

### 7.4 应该停止的情况

以下情况通常应该停止：

1. 用户目标已经满足
2. 当前结果已经足够回答用户
3. 缺少必要信息且无法继续
4. 达到最大步数或重试上限
5. 连续失败且没有合理替代方案

### 7.5 为什么“够好就停”很重要

真实系统里，不要盲目追求“完美才停”。  
否则常常会导致：

- 调用成本变高
- 响应速度变慢
- 死循环风险上升

工程上更常见的原则是：

`结果已经足够满足当前用户需求，就可以停。`

### 7.6 停止条件通常来自哪里

#### 模型判断

模型根据工具结果做语义判断：

- 是否已经满足用户目标
- 是否还需下一步

#### 系统硬规则

系统设定边界：

- 最多几步
- 最多重试几次
- 同一工具失败几次就停止

最稳定的做法是两者结合：

`模型负责语义判断，系统负责边界兜底。`

### 7.7 常见误区

#### 误区 1：能继续做就继续做

不对。  
Agent 不应该无意义地多做步骤。

#### 误区 2：只靠模型自己判断停止就够了

不对。  
没有系统级兜底时，容易出现死循环和过度调用。

### 7.8 课后问答

#### Q1：为什么停止条件是 Agent 的核心能力？

答：

因为 Agent 是动态执行系统。  
只要是动态执行，就必须判断：

- 当前是否够了
- 是否还值得继续
- 是不是已经应该结束

#### Q2：追问用户算不算一种“停止当前循环”？

答：

算。  
当缺少关键参数时，最合理的做法往往不是盲猜或无脑重试，而是暂停当前执行，向用户补问必要信息。

## 8. 第六课：什么时候该补问用户，什么时候该自己继续

### 8.1 本课目标

理解 Agent 的自主性不是“什么都猜”，也不是“什么都问”，而是合理划分边界。

### 8.2 最核心的一句话

`能自己完成的就自己完成，缺关键条件时再问用户。`

### 8.3 应该自己继续的情况

以下情况通常不需要追问：

1. 用户意图已经足够明确
2. 缺失信息可以从上下文推断
3. 系统已有合理默认值
4. 当前工具结果已经足够支持下一步

例如：

- 用户上一轮说自己在北京，这一轮问“那明天会下雨吗”
- 用户明确说“读取 demo.py 并简单总结”

### 8.4 必须补问用户的情况

以下情况通常必须补问：

1. 缺少关键参数
2. 存在多个合理解释，不能随便猜
3. 用户偏好会显著影响任务结果
4. 猜错代价较高

例如：

- “帮我查天气”但没有地点
- “帮我看文件”但目录下有很多文件

### 8.5 默认值和必须确认信息的边界

#### 更适合默认的信息

- 输出语言
- 温度单位
- 展示风格
- 默认时间范围

#### 更适合补问的信息

- 地点
- 文件名
- 查询对象
- 关键日期

一句话：

`影响任务对象的信息，优先补问；影响展示形式的信息，可以默认。`

### 8.6 为什么追问不是失败

很多新手误以为 Agent 一追问就说明不够聪明。  
实际上，合理追问反而说明系统知道：

- 什么时候信息不够
- 什么时候不能乱猜
- 怎么把错误成本控制住

### 8.7 常见误区

#### 误区 1：强模型就应该尽量少问用户，缺信息自己猜

不对。  
模型越强，越应该知道什么时候该确认，而不是盲猜。

#### 误区 2：为了避免出错，什么都问用户

也不对。  
这样会让 Agent 丧失“替用户省判断成本”的价值。

### 8.8 课后问答

#### Q1：什么样的信息算“关键信息”？

答：

会改变任务对象、显著影响结果、或者猜错代价高的信息，就属于关键信息。

#### Q2：怎么判断当前是否应该补问？

答：

可以问自己一句：

`如果我现在不问用户直接继续，结果大概率会不会偏？`

- 如果不会偏，可以继续
- 如果很可能偏，就应该补问

## 9. 第七课：短期记忆、长期记忆、RAG 到底怎么区分

### 9.1 本课目标

彻底分清：

- 短期记忆
- 长期记忆
- RAG

避免把“聊天记录”“用户偏好”“外部知识库”混成一锅。

### 9.2 三句话总纲

- `短期记忆`：为了当前会话不断线
- `长期记忆`：为了跨会话不失忆
- `RAG`：为了从外部知识源补资料

### 9.3 什么是短期记忆

短期记忆保存的是当前会话里临时有用的信息，例如：

- 用户刚刚说自己在杭州
- 上一轮提到的是 `demo.py`
- 当前任务已经调过一次天气工具
- 上一步工具调用失败过一次

它更像当前工作台上的便签。

### 9.4 什么是长期记忆

长期记忆保存跨会话仍有价值的信息，例如：

- 用户喜欢简洁回答
- 用户常住上海
- 用户经常问 Java 架构问题
- 用户偏好中文输出

它更像用户档案或长期偏好记录。

### 9.5 什么是 RAG

RAG 本质上是从外部资料中检索知识。  
例如：

- 公司制度文档
- 产品说明书
- FAQ
- 内部知识库

它解决的是“模型知识不够或知识过时”的问题。

### 9.6 为什么 RAG 不等于 Memory

这是最关键的区分。

#### Memory 关注的是

- 我们之前聊过什么
- 用户偏好是什么
- 当前任务相关信息是什么

#### RAG 关注的是

- 外部文档里写了什么
- 知识库里有什么资料
- 某条业务规则或说明在哪里

一句话：

`Memory 是记住我们，RAG 是查外部世界。`

### 9.7 存储手段和概念本质不要混

即使把“用户喜欢简洁回答”存进向量库，它仍然是长期记忆，不是 RAG。

判断标准不是“存在哪”，而是“存的是什么”。

### 9.8 常见误区

#### 误区 1：只要是长期保存的东西，就是长期记忆

不对。  
如果保存的是外部资料，那更接近 RAG，不是 Memory。

#### 误区 2：接了向量库就一定有记忆能力

不对。  
向量库也可能只是用来做文档检索。

### 9.9 课后问答

#### Q1：什么叫短期记忆？

答：

当前会话和当前任务中临时有用的信息。

#### Q2：什么叫长期记忆？

答：

跨会话仍然有价值的用户信息、偏好或稳定事实。

#### Q3：什么叫 RAG？

答：

从外部知识库中检索资料来补充模型知识。

#### Q4：如果“用户喜欢简洁回答”被存进向量库，它算长期记忆还是 RAG？

答：

算长期记忆。  
因为关键看信息语义，而不是存储介质。

## 10. 第八课：什么叫 State，为什么 State 不等于 Memory

### 10.1 本课目标

理解：

- `history`
- `memory`
- `state`

三者不是一回事。

### 10.2 一句话总纲

- `history` 是记录
- `memory` 是保留
- `state` 是控制

### 10.3 什么是 State

State 指的是当前这一刻任务执行到哪里了。

例如：

- 当前任务是“读取并总结文件”
- 目标文件是 `demo.py`
- 读文件步骤已完成
- 总结步骤未开始
- 当前工具已重试 2 次
- 当前循环已经执行了 1 步

这些信息的共同特点是：

- 和当前流程直接相关
- 用来决定下一步该做什么
- 常常是临时性的

### 10.4 什么是 Memory

Memory 更关注“要记住什么”，例如：

- 用户喜欢简洁回答
- 用户常住上海
- 用户上一轮提到某个项目名

Memory 更偏信息保留和复用。

### 10.5 什么是 History

History 更接近聊天原文，例如：

- 用户说了什么
- 系统回复了什么

它是最原始的对话流水。

### 10.6 State 和 Memory 的本质区别

#### State 解决的问题

- 下一步做什么
- 当前做到哪了
- 是否已重试
- 是否该停止

#### Memory 解决的问题

- 上下文别断
- 用户偏好别丢
- 有价值的信息以后还能用

一句话：

- `Memory 负责别忘`
- `State 负责别乱`

### 10.7 为什么 State 在 Agent 中很重要

因为 Agent 往往是多步执行的。  
只要任务不是一步结束，就需要 State 记录：

- 走到哪一步了
- 还剩哪一步
- 是否失败过
- 是否该继续

没有 State，系统很难稳定地跑多步任务。

### 10.8 常见误区

#### 误区 1：只要被存下来就是 Memory

不对。  
History 和 State 也可以被存下来，但职责不一样。

#### 误区 2：State 就是聊天记录

不对。  
聊天记录是 History，State 是当前执行控制信息。

### 10.9 课后问答

#### Q1：最近三轮对话原文属于什么？

答：

属于 `history`，因为它是原始聊天记录。

#### Q2：用户长期喜欢简洁回答属于什么？

答：

属于 `memory`，因为它是长期可复用的信息。

#### Q3：当前任务已完成读文件，但未开始总结，属于什么？

答：

属于 `state`，因为它描述的是当前执行进度。

## 11. 第九课：为什么 Orchestrator 往往比 LLM 更能决定 Agent 的稳定性

### 11.1 本课目标

从“懂概念”进入“会判断系统质量”。

### 11.2 最核心的一句话

`LLM 决定 Agent 的上限，Orchestrator 决定 Agent 的下限。`

### 11.3 什么叫“LLM 决定上限”

模型越强，通常越擅长：

- 理解复杂问题
- 正确选择工具
- 理解 observation
- 生成更自然、更高质量的最终回复

所以模型决定的是：

`Agent 能聪明到什么程度。`

### 11.4 什么叫“Orchestrator 决定下限”

Orchestrator 负责的是系统控制和边界兜底，例如：

- 组织输入上下文
- 调用模型
- 解析模型输出
- 判断是否工具调用
- 校验参数
- 执行工具
- 处理异常
- 控制最大步数
- 控制重试次数
- 判断停止条件
- 写入 state/memory

这些事情大多直接影响稳定性。

所以 Orchestrator 决定的是：

`最差情况下，系统会糟到什么程度。`

### 11.5 为什么很多问题不能第一反应怪模型

因为很多 Agent 的问题其实出在：

- Prompt 设计不清楚
- 工具描述不清楚
- 参数设计不合理
- 输出解析不严谨
- 状态没有维护好
- 重试机制失控
- 停止条件不明确

这些问题往往都属于系统设计问题，而不是模型智力问题。

### 11.6 为什么模型强也不能替代 Orchestrator

模型擅长的是：

- 语义理解
- 推理
- 决策建议

但天然不保证：

- 参数一定合法
- 工具一定不会乱调
- 不会死循环
- 失败一定会被正确处理

这些更像是流程控制和系统约束问题。

### 11.7 稳定 Agent 常见的 Orchestrator 责任

典型包括：

1. 输入控制
2. 输出解析
3. 工具执行控制
4. 循环控制
5. 停止控制
6. 失败后的兜底

### 11.8 语义判断和边界兜底的分工

一个很重要的理解是：

`LLM 负责理解，Orchestrator 负责约束。`

例如：

- 是否语义上满足用户目标，偏 LLM
- 是否超过步数、是否需要硬停止，偏 Orchestrator

很多场景不是单方独立完成，而是两者共同作用。

### 11.9 常见误区

#### 误区 1：模型够强，系统自然就稳

不对。  
模型强只代表上限高，不代表不会乱。

#### 误区 2：Orchestrator 只是个胶水层

不对。  
它实际上决定了 Agent 的流程控制质量和稳定性下限。

### 11.10 课后问答

#### Q1：如果模型输出了错误格式的工具调用，应该由谁兜底？

答：

主要应该由 Orchestrator 兜底。  
因为这是流程控制和格式容错问题。

#### Q2：同一工具连续失败 2 次后停止，这属于谁的职责？

答：

属于 Orchestrator 的职责。  
因为这是典型的系统边界规则。

#### Q3：根据工具结果判断任务是否已完成，这属于谁的职责？

答：

两者共同作用。  
LLM 更偏语义判断，Orchestrator 更偏边界和规则控制。

## 12. 总复习：关键概念一页总结

### 12.1 Agent 的本质

Agent 是围绕任务目标运行的闭环系统，而不是单次工具调用。

### 12.2 最小闭环

`用户输入 -> 模型决策 -> 工具执行 -> 观察结果 -> 再决策 -> 最终回复`

### 12.3 核心模块

- `LLM`：负责思考和决策
- `Prompt`：负责行为规则
- `Tool`：负责外部执行能力
- `Orchestrator`：负责流程控制
- `Memory`：负责信息保留
- `State`：负责执行状态
- `RAG`：负责外部知识检索

### 12.4 核心分工

- LLM 负责想
- Tool 负责做
- Prompt 负责约束怎么想
- Orchestrator 负责把流程串起来
- Memory 负责别忘
- State 负责别乱
- RAG 负责查外部知识

### 12.5 重点区分

#### History vs Memory vs State

- `History`：原始聊天记录
- `Memory`：提炼后的可复用信息
- `State`：当前流程控制信息

#### 短期记忆 vs 长期记忆 vs RAG

- `短期记忆`：当前会话里临时有用的信息
- `长期记忆`：跨会话仍有价值的稳定信息
- `RAG`：从外部知识库查资料

### 12.6 工程原则

- 不是工具越多越好，而是工具定义越清楚越好
- 不是参数越全越好，而是越清晰越稳定
- 不是步骤越多越好，而是够用就停
- 不是越少追问越好，而是该问时问得准
- 不是模型越强系统就越稳，而是 Orchestrator 要把下限兜住

## 13. 面试/复述速记版

以下内容适合口头表达、复习、面试前速记。

### 13.1 什么是 Agent

Agent 不是简单会调用工具的大模型，而是围绕任务目标运行的闭环系统。  
它会根据当前状态做决策，调用工具与外部环境交互，再根据结果继续执行或停止，最终输出用户可理解的结果。

### 13.2 Agent 和普通聊天模型的区别

普通聊天模型主要是一问一答。  
Agent 除了生成内容，还要：

- 做决策
- 调工具
- 评估结果
- 多步循环
- 管理状态

### 13.3 Prompt 的作用

Prompt 在 Agent 中不是简单的人设文案，而是自然语言形式的控制逻辑。  
它定义模型是谁、目标是什么、有哪些工具、什么时候该调工具、结果回来后如何处理、最终输出格式是什么。

### 13.4 Tool 的作用

Tool 是 Agent 与外部世界交互的能力。  
LLM 决定用哪个工具，工具负责真正执行，工具结果再回给模型做解释和下一步决策。

### 13.5 为什么工具描述和参数设计重要

工具描述本质上是给模型看的 API 文档，帮助模型正确选工具。  
参数设计决定模型是否能稳定地把工具用对，因此直接影响 Agent 稳定性。

### 13.6 Memory、State、RAG 的区别

- Memory 是记住用户和任务相关的可复用信息
- State 是记录当前执行流程的状态和进度
- RAG 是从外部知识库检索资料补充模型知识

### 13.7 为什么 Orchestrator 很重要

LLM 决定上限，Orchestrator 决定下限。  
模型决定系统能多聪明，Orchestrator 决定系统能否稳定运行，包括格式解析、工具执行、异常处理、重试控制、停止条件等。

### 13.8 五句最值得背的话

1. Agent 的本质是目标驱动的闭环。
2. Prompt 是自然语言形式的控制逻辑。
3. 工具提供观察，模型负责解释和下一步决策。
4. Memory 负责别忘，State 负责别乱。
5. LLM 决定上限，Orchestrator 决定下限。
